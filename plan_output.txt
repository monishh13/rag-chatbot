1. Project Overview
Objective
Build a fully local Retrieval-Augmented Generation (RAG) chatbot capable of answering
employee questions using internal company PDF documents.
The system:
∩é╖ ingests PDF documents,
∩é╖ converts them into semantic vector embeddings,
∩é╖ stores them in a local vector database,
∩é╖ retrieves the most relevant text fragments,
∩é╖ and uses an LLM to generate grounded responses.
The chatbot must:
∩é╖ minimize hallucinations,
∩é╖ operate with deterministic outputs,
∩é╖ support source traceability,
∩é╖ and maintain low infrastructure complexity.

3. Technology Stack & Engineering
Justifications
Backend Framework
FastAPI
Why FastAPI?
The chat pipeline is highly I/O-bound:
∩é╖ embedding generation,
∩é╖ vector retrieval,
∩é╖ external LLM streaming.
FastAPI provides:
∩é╖ asynchronous request handling,
∩é╖ automatic OpenAPI/Swagger documentation,
∩é╖ strong typing with Pydantic,
∩é╖ lower latency under concurrent workloads.
Review Defense
ΓÇ£FastAPI was selected because asynchronous request handling prevents blocking during
concurrent LLM and embedding operations. The framework also provides automatic API
documentation and schema validation, reducing backend maintenance complexity.ΓÇ¥
Vector Database
ChromaDB
Why ChromaDB?
Advantages:
--- PAGE 3 ---
∩é╖ persistent local storage,
∩é╖ metadata filtering,
∩é╖ built-in document association,
∩é╖ zero cloud dependency,
∩é╖ simple deployment.
Review Defense
ΓÇ£ChromaDB provides persistent vector storage with integrated metadata management. Unlike
raw FAISS implementations, it removes the need for maintaining external mapping layers
between vectors and document chunks.ΓÇ¥
Embedding Model
Sentence Transformers
Model: all-MiniLM-L6-v2
Why This Model?
Advantages:
∩é╖ lightweight,
∩é╖ runs fully local,
∩é╖ low latency,
∩é╖ excellent semantic retrieval for small corpora,
∩é╖ deterministic behavior.
Review Defense
ΓÇ£The embedding model was intentionally selected for low inference latency and local
execution. For a small enterprise corpus, external embedding APIs would introduce
unnecessary cost and additional network failure points.ΓÇ¥
LLM Provider
Anthropic
Model: Claude 3.5 Sonnet
Why Claude?
Advantages:
--- PAGE 4 ---
∩é╖ strong instruction adherence,
∩é╖ reliable summarization,
∩é╖ reduced hallucination tendency,
∩é╖ strong context utilization.
Review Defense
ΓÇ£Claude was selected due to its strong instruction-following capabilities and reliable
grounded response generation in retrieval-augmented pipelines.ΓÇ¥
4. Functional Requirements
The system must:
∩é╖ ingest multiple PDFs,
∩é╖ extract text from PDFs,
∩é╖ split text into semantic chunks,
∩é╖ generate vector embeddings,
∩é╖ store embeddings persistently,
∩é╖ retrieve top-k relevant chunks,
∩é╖ filter low-confidence retrievals,
∩é╖ generate grounded answers,
∩é╖ return source references,
∩é╖ stream responses to frontend,
∩é╖ handle invalid inputs gracefully.
5. Non-Functional Requirements
Scalability
The architecture must:
∩é╖ support modular component replacement,
∩é╖ allow future migration to cloud vector databases,
∩é╖ support increasing document counts.
Reliability
The system must:
--- PAGE 5 ---
∩é╖ use deterministic generation (temperature=0),
∩é╖ support persistent vector storage,
∩é╖ prevent hallucinated responses when retrieval confidence is low.
Security
The system must:
∩é╖ protect API keys via environment variables,
∩é╖ restrict production CORS domains,
∩é╖ avoid exposing raw internal documents.
Maintainability
The architecture must separate:
∩é╖ ingestion logic,
∩é╖ retrieval logic,
∩é╖ API logic,
∩é╖ frontend UI.

Engineering Justification
ΓÇ£Recursive splitting preserves paragraph and sentence boundaries before falling back to raw
character segmentation.ΓÇ¥
Step 3 ΓÇö Generate Embeddings
Use batched embedding generation:

Engineering Justification
ΓÇ£Batch vectorization improves CPU throughput versus sequential embedding execution.ΓÇ¥
--- PAGE 8 ---
Step 4 ΓÇö Store in ChromaDB
Store:
∩é╖ chunk text,
∩é╖ metadata,
∩é╖ ids,
∩é╖ embeddings.
Metadata Structure
{
"source": doc_name
}
Do NOT duplicate chunk text inside metadata.
Step 5 ΓÇö Idempotent Ingestion
Prevent duplicate insertion:
existing = collection.get(ids=[chunk_id])
if existing["ids"]:
continue
Engineering Justification
ΓÇ£Idempotent ingestion prevents duplicate vector accumulation during repeated indexing
operations.ΓÇ¥
Phase 3 ΓÇö Retrieval Pipeline

Step 3 ΓÇö Similarity Threshold Filtering
Purpose
Prevent low-confidence chunks from polluting the prompt.
THRESHOLD = 1.2
Filter irrelevant retrievals before prompt assembly.
Engineering Justification
ΓÇ£Similarity threshold filtering prevents semantically weak retrieval fragments from
contaminating generation context.ΓÇ¥
